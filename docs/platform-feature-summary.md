# Reading Coach Lab 구현 기능 정리

작성 기준: 2026-07-01 현재 코드 기준

이 문서는 현재 플랫폼에 실제로 구현된 기능을 연구 운영 관점에서 정리한 것이다. 칸아카데미 Writing Coach를 참고한 화면 흐름이 있지만, 현재 플랫폼의 핵심 목적은 비문학 글쓰기 과제를 수행하는 동안 학생과 AI의 대화, 개요 작성, 초안 작성, 수정, 제출 과정을 데이터셋 형태로 수집하는 연구 파일럿이다.

## 1. 플랫폼의 현재 성격

현재 플랫폼은 다음 목적을 갖는 단일 웹 애플리케이션이다.

- 교사가 비문학 지문과 글쓰기 문제를 과제로 만들고 특정 반에 배정한다.
- 학생은 배정된 과제를 보고 읽기, 개요 작성, 초안 쓰기, 고쳐쓰기, 제출 단계를 진행한다.
- 학생은 단계 중간에 AI 코치와 멀티턴 대화를 나눌 수 있다.
- AI 코치는 답을 대신 쓰기보다 질문, 근거 점검, 과제 복귀, 고쳐쓰기 안내를 하도록 제한되어 있다.
- 학생의 대화, 개요, 초안, 붙여넣기, 피드백 확인, 최종 제출, 교사 검토가 세션 로그로 저장된다.
- 저장된 로그는 JSON 데이터셋과 라벨링용 CSV로 내보낼 수 있다.

현재 구현은 연구용 파일럿에 가깝다. 실제 학교 운영용 LMS, 완전한 인증 시스템, 자동 라벨링 시스템, 실험 조건 배정 시스템까지 완성된 상태는 아니다.

## 2. 접근 경로와 역할

플랫폼은 브라우저 경로에 따라 주요 화면을 연다.

| 경로 | 화면 | 주 사용자 | 현재 기능 |
| --- | --- | --- | --- |
| `/` | 과제 목록/교사 홈 | 교사, 연구자 | 과제 탐색, 검색, 필터, 미리보기, 배정, 학생 현황, 로그, 계정 관리 이동 |
| `/assignments/new` | 과제 만들기/수정 | 교사 | 비문학 지문, 문제, 요구사항, 반 배정, 일정 입력 |
| `/student` | 학생 과제 또는 작업 화면 | 학생 | 배정 과제 목록, 과제 시작, 읽기/개요/초안/고쳐쓰기 진행 |
| `/review` | 학생 현황 | 교사 | 학생별 진행 상태와 과정 로그 확인, 교사 검토 메모 |
| `/export` | 연구 로그 | 교사, 연구자 | JSON/CSV 내보내기, 스키마/코드북 다운로드 |
| `/accounts` | 계정 관리 | 교사, 연구자 | 반, 학생, 교사 계정 생성/삭제, 학생 일괄 생성 |

역할은 `teacher`와 `student` 두 가지로 나뉜다.

- 학생은 참여자 코드 또는 학생 아이디/비밀번호로 들어간다.
- 교사는 교사 아이디/비밀번호로 들어간다.
- 기본 교사 계정은 `test / test`이다.
- 샘플 학생은 `minseo / MINSEO-2026`, `joon / JOON-2026`이며 참여자 코드는 각각 `S-MINSEO`, `S-JOON`이다.
- 학생 화면에서는 역할 전환 버튼이 보이지 않고, 로그아웃으로 다른 계정 진입이 가능하다.
- 교사 전용 경로에 학생 또는 미로그인 사용자가 접근하면 로그인 화면이 먼저 나온다.

## 3. 과제 관리 기능

교사 홈에서는 과제를 목록 형태로 관리한다.

구현된 기능은 다음과 같다.

- 과제 목록 표시
- 현재 선택된 과제 강조
- 과제별 제목, 문제, 학년/난이도, 글 유형, 반, 진행 요약 표시
- 과제별 배정 학생 수, 진행 중 학생 수, 제출 완료 학생 수 표시
- 검색어 기반 과제 검색
- 글 유형 범주 필터
- 학년/난이도 필터
- 과제 미리보기
- 미리보기 화면에서 배정할 반 선택
- 선택한 반에 과제 배정
- 기존 과제 수정 진입
- 새 과제 만들기 진입
- 학생 현황, 학생 화면 보기, 로그 보기, 계정 관리 화면으로 이동

과제 만들기/수정 화면에서 설정할 수 있는 항목은 다음과 같다.

- 과제 제목
- 과제 유형
  - `전체 글쓰기 과정`: 이해, 개요, 초안, 고쳐쓰기를 모두 기록
  - `초안 피드백과 수정`: 이미 쓴 글을 가져와 피드백과 수정 기록을 보는 목적의 유형
- 학년 또는 난이도
- 글 유형
- 목표 분량
- 최소 글자 수
- 비문학 지문
- 해결할 문제
- 학생에게 보일 요구사항
- 근거와 출처 안내
- 배정할 반
- 시작일, 시작 시간
- 마감일, 마감 시간

현재 `초안 피드백과 수정` 유형은 과제 속성으로 저장되지만, 학생 화면에서 전체 글쓰기 과정과 완전히 다른 별도 UX로 분기되는 수준까지는 구현되어 있지 않다.

## 4. 계정과 반 관리 기능

계정 관리 화면에서는 반, 학생, 교사 계정을 만들고 삭제할 수 있다.

반 만들기:

- 새 반 이름 입력
- 담당 교사 선택
- 반 생성
- 반 목록에서 담당 교사와 학생 수 확인
- 휴지통 버튼으로 반 삭제

학생 만들기:

- 학생 반 선택
- 학생 번호 입력
- 참여자 코드 입력
- 학생 아이디 입력
- 학생 비밀번호 입력
- 학생 이름 입력
- 학생 생성
- 학생 계정 목록에서 반, 번호, 이름, 참여자 코드, 아이디, 초기 비밀번호 확인
- 휴지통 버튼으로 학생 삭제

학생 일괄 만들기:

- 일괄 생성 반 선택
- 시작 번호 입력
- 생성 갯수 입력
- 아이디 접두어 입력
- 비밀번호 접두어 입력
- 학생 이름 접두어 입력
- 예를 들어 접두어 `student-`, 시작 번호 `3`, 생성 갯수 `5`이면 `student-03`부터 `student-07`까지 생성
- 참여자 코드는 생성된 아이디를 대문자로 바꾼 값으로 저장

교사 만들기:

- 교사 이름 입력
- 교사 아이디 입력
- 교사 비밀번호 입력
- 교사 생성
- 교사 목록에서 아이디와 비밀번호 확인
- 휴지통 버튼으로 교사 삭제

주의할 점:

- 현재 비밀번호는 연구 파일럿 편의를 위해 화면과 로컬 상태에 평문으로 보인다.
- export JSON에서는 교사/학생 비밀번호가 제외된다.
- 운영용 보안 인증 시스템이나 서버 DB 기반 계정 관리는 아직 아니다.

## 5. 학생 과제 수행 흐름

학생이 로그인하면 자신이 속한 반에 배정된 과제만 볼 수 있다. 과제 목록에는 과제 제목, 문제, 난이도, 목표 분량, 상태, 마감 정보, 제출 요구사항, 지문 미리보기가 표시된다.

학생 작업 화면은 네 단계로 구성되어 있다.

### 5.1 읽기 단계

학생은 과제 지문과 문제를 읽는다.

구현된 기능:

- 현재 과제 제목 표시
- `과제 보기` 버튼으로 과제 참조 모달 열기
- 단계 표시
- 다음 단계 이동
- AI 코치 패널에서 과제 설명, 요구사항 확인 등 빠른 질문 사용

### 5.2 개요 작성 단계

학생은 최종 글을 쓰기 전에 생각을 구조화한다.

구현된 입력 항목:

- 중심 생각
- 근거 또는 예시 여러 개
- 출처 메모 여러 개
- 설명 또는 연결
- 반대 의견

구현된 조작:

- 근거 추가
- 근거별 휴지통 삭제
- 출처 추가
- 출처별 휴지통 삭제
- 개요 점검
- 초안 쓰기 시작

개요 점검은 다음 조건을 확인한다.

- 중심 생각이 충분한가
- 근거가 두 개 이상 있는가
- 출처 메모가 있는가
- 근거와 주장을 잇는 설명이 충분한가
- 반론이 있는가

조건이 부족하면 경고가 표시되고, 그래도 초안으로 넘어갈 수 있는 버튼이 제공된다. 이 경고 자체도 로그 이벤트로 남는다.

### 5.3 초안 쓰기 단계

학생은 개요를 바탕으로 직접 초안을 작성한다.

구현된 기능:

- 초안 입력창
- 글자 수 확인
- 개요 요약 패널
- AI 코치와 대화
- 붙여넣기 이벤트 기록
- 고쳐쓰기 단계로 이동

붙여넣기는 완전히 차단하지 않고, 붙여넣기 시도를 기록한다. 기록되는 값은 붙여넣은 텍스트 길이, 줄 수, 앞 80자 미리보기, 시간, 대상 필드이다.

### 5.4 고쳐쓰기/제출 단계

학생은 피드백을 보고 초안을 직접 고친 뒤 제출한다.

구현된 기능:

- 피드백 목록 표시
- 선택한 피드백 초점 표시
- 초안 편집창에서 계속 수정
- `내 수정 확인` 기능
- 제안 해결 표시
- 최종 제출
- 제출 완료 표시

피드백은 기본 규칙 기반 제안이 먼저 표시되고, Gemini API가 연결되어 있으면 AI 제안으로 갱신된다. 단, 학생 대신 문장을 직접 고쳐주는 방식은 아니다.

## 6. AI 코치 기능

AI 코치는 학생의 답을 대신 작성하지 않고, 과제와 지문 안에서 질문으로 돕도록 설계되어 있다.

현재 응답 유형은 여섯 가지이다.

| 응답 유형 | 의미 |
| --- | --- |
| `clarify` | 과제나 요구사항을 설명하고 학생이 다시 정리하게 함 |
| `question` | 주장, 이유, 반론을 스스로 생각하도록 질문 |
| `evidence_check` | 근거와 출처, 주장과의 연결을 점검하도록 질문 |
| `redirect` | 과제와 무관한 대화를 다시 과제로 돌림 |
| `revision_guidance` | 초안의 약한 부분을 스스로 찾도록 안내 |
| `refusal` | 답안 작성, 문장 고쳐주기, 대신 쓰기 요청을 거절 |

현재 서버 정책:

- `READING_COACH_AI_MODE=mock`이면 로컬 규칙 기반 코치가 응답한다.
- mock이 아니면 Gemini API를 호출한다.
- Gemini 모델은 환경 변수 `GEMINI_MODEL`을 사용하며 기본값은 `gemini-3.5-flash`이다.
- `GEMINI_API_KEY`가 없으면 API 요청은 실패한다.
- Gemini 호출 시 `temperature`는 `0.2`로 설정되어 있다.
- Gemini 응답은 JSON 형식이어야 하며, 응답 유형도 위 여섯 가지 중 하나여야 한다.
- 과제 밖 질문(`날씨`, `축구`, `게임`, `노래`, `뉴스` 등)과 대신 써달라는 요청은 Gemini 호출 전에 로컬 정책으로 먼저 차단된다.

현재 코치 프롬프트의 핵심 지시:

- 초등 고학년 학생을 돕는 한국어 글쓰기 코치로 행동한다.
- 학생 대신 답안, 문단, 문장, 결론을 써주지 않는다.
- 학생의 주장, 근거, 반론을 스스로 고르게 질문한다.
- 과제와 지문 밖의 일반 대화는 과제로 되돌린다.
- 아첨하지 않고 학생 말에 무조건 동의하지 않는다.
- 응답은 짧고 구체적으로 한다.

이 구조 덕분에 AI 응답마다 `responseType`이 남는다. 다만 `responseType`은 시스템 정책 분류일 뿐이고, 실제 비판적 사고, 인지 외주화, 아첨 여부를 자동 판정하는 라벨은 아직 아니다.

## 7. 피드백과 수정 확인 기능

고쳐쓰기 단계에는 두 종류의 피드백 생성 방식이 있다.

규칙 기반 기본 피드백:

- 주장이 초안에 잘 드러나는지 확인
- 개요에 쓴 근거가 초안에 들어갔는지 확인
- 출처 표시가 있는지 확인
- 반론 뒤에 다시 주장으로 돌아오는 연결이 있는지 확인
- 글이 너무 짧은지 확인
- 문제가 거의 없으면 전체 흐름 점검 제안 표시

Gemini 기반 피드백:

- 초안을 대신 고쳐 쓰지 않고 학생이 직접 고칠 위치와 이유만 제안
- 최대 4개의 제안 생성
- 범주는 `주장과 초점`, `근거와 설명`, `구조와 흐름`, `문장 표현`, `좋은 점검` 중 하나

수정 확인:

- 학생이 특정 제안을 선택하고 `내 수정 확인`을 누르면 해당 제안이 해결됐는지 판단한다.
- 기본 규칙 기반 제안은 결정적 규칙으로 확인한다.
- AI가 만든 비기본 제안은 Gemini에게 해결 여부와 짧은 이유를 요청할 수 있다.
- 해결되면 `suggestion_checked`, `suggestion_resolved` 이벤트가 남는다.

## 8. 복사와 붙여넣기 관련 구현

현재 구현은 다음과 같다.

- AI 코치 패널과 채팅 로그 영역에는 `copy` 이벤트 방지가 걸려 있다.
- 초안 입력창과 고쳐쓰기 입력창의 붙여넣기는 기록된다.
- 붙여넣기 자체를 완전히 막지는 않는다.
- 붙여넣기 기록에는 전체 원문이 아니라 길이, 줄 수, 앞 80자 미리보기만 저장된다.

따라서 현재 정책은 “AI 답변 복사 방지 + 외부 붙여넣기 감시”에 가깝다. 모든 기기와 브라우저 상황에서 복사/붙여넣기를 완전히 봉쇄하는 보안 기능으로 보기는 어렵다.

## 9. 교사 현황과 과정 검토 기능

교사 현황 화면에서는 학생별 과제 진행 상태와 과정 기록을 확인할 수 있다.

구현된 기능:

- 학생 검색
- 상태 필터
  - 전체
  - 시작 전
  - 진행 중
  - 제출 완료
- 검토 상태 필터
  - 전체
  - 미검토
  - 추가 확인 필요
  - 검토 완료
- 학생별 번호, 참여자 코드, 진행 상태 표시
- 학생 과정 보기
- 대화, 개요, 초안, 붙여넣기, 제출 기록 확인
- 교사 검토 상태와 메모 저장

교사 검토 메모는 세션 내부의 `teacherReview`에 저장되고, `teacher_review_updated` 이벤트로도 남는다.

## 10. 데이터 저장 방식

현재 데이터는 두 방식으로 저장된다.

브라우저 저장:

- `localStorage` 키: `reading-coach-lab:v1`
- 계정, 반, 과제, 세션, 현재 선택 역할, 현재 활성 과제/세션을 저장
- 같은 브라우저/같은 origin 기준으로 유지된다.

서버 파일 저장:

- 상태가 바뀔 때 `/api/pilot-state`로 전체 상태를 POST한다.
- Vite 개발 서버가 `.omo/evidence/khan-parity-auth-persistence/server-state.json`에 저장한다.
- 파일 저장 성공/실패 상태는 export 화면에 표시된다.

주의:

- 현재는 개발 서버 기반 파일 저장이다.
- 여러 기기, 여러 브라우저, 실제 배포 서버에서 동시 접속을 안정적으로 처리하는 DB 구조는 아니다.
- 연구 파일럿에서 데이터를 안전하게 수집하려면 별도 서버 저장소와 백업 정책이 필요하다.

## 11. 데이터셋 Export 기능

연구 로그 화면에서는 현재 상태를 연구용 파일로 내보낼 수 있다.

제공되는 다운로드:

- `reading-coach-pilot-dataset.json`
- `reading-coach-labeling-rows.csv`
- `pilot-dataset.schema.json`
- `labeling-codebook.md`
- `data-dictionary.md`

JSON 데이터셋에는 다음이 포함된다.

- 스키마 버전
- 교사 계정 목록
- 학생 계정 목록
- 반 목록
- 과제 목록
- 세션 목록
- 선택된 역할
- 활성 과제/세션
- 앱 메타데이터
- export 메타데이터
- 파일 저장 상태

비밀번호 처리:

- 내부 상태에는 학생/교사 비밀번호가 있다.
- export JSON에서는 교사와 학생 비밀번호가 제외된다.
- 학생 이름, 로그인 아이디, 참여자 코드는 export에 포함되므로 실제 연구에서는 별도 익명화 정책이 필요하다.

세션 데이터에는 다음이 포함된다.

- 세션 ID
- 학생 익명 ID
- 학생 계정 ID와 이름
- 과제 정보
- 현재 단계
- 이벤트 로그
- 채팅 턴
- 개요 스냅샷
- 초안 스냅샷
- 붙여넣기 이벤트
- 최종 제출
- 교사 검토 메모
- LLM 모드와 모델

라벨링 CSV는 세션 이벤트를 행 단위로 펼친 것이다.

CSV 열:

- `sessionId`
- `studentAnonymousId`
- `assignmentId`
- `turnOrEventId`
- `timestamp`
- `stage`
- `speaker`
- `criticalThinkingLabel`
- `offloadingLabel`
- `sycophancyLabel`
- `evidenceText`
- `raterNotes`

현재 라벨 값은 모두 `none`으로 초기화된다. 즉, 라벨링용 틀은 구현되어 있지만 전문가 라벨링이나 LLM judge 자동 라벨링은 아직 구현되어 있지 않다.

## 12. 라벨링 코드북과 연구 신호

현재 코드북에는 세 가지 라벨 축이 고정되어 있다.

비판적 사고 라벨:

- `evidence_request`
- `source_verification`
- `counterargument_exploration`
- `alternative_comparison`
- `claim_revision`
- `uncertainty_acknowledgment`

인지 외주화 라벨:

- `answer_delegation`
- `judgment_delegation`
- `unreviewed_acceptance`
- `verification_avoidance`
- `counterargument_avoidance`
- `completion_priority`

아첨 라벨:

- `unsupported_agreement`
- `overpraise`
- `missed_correction`
- `balanced_challenge`

현재 플랫폼에서 연구 신호로 활용 가능한 데이터는 다음과 같다.

- 학생이 AI에게 보낸 질문
- AI의 응답 텍스트와 응답 유형
- 과제 밖 대화 시도
- 대신 써달라는 요청과 거절 여부
- 개요의 주장, 근거, 출처, 설명, 반론 변화
- 초안 작성 과정의 스냅샷
- 붙여넣기 시도
- 피드백 생성과 피드백 확인
- 수정 확인 결과
- 최종 제출문
- 교사 검토 메모

이 데이터는 “최종 글의 품질”만 보는 것이 아니라, 학생이 어떤 순간에 판단하고, 검증하고, 회피하고, AI에게 넘겼는지를 분석하는 기반이 된다.

## 13. 현재 구현으로 가능한 연구

현재 상태만으로 가능한 파일럿 연구는 다음과 같다.

학생-AI 대화 과정 수집:

- 과제별 학생 대화 로그 수집 가능
- 학생별 세션 구분 가능
- AI 응답 유형 기록 가능
- 대화와 개요/초안/제출 연결 가능

인지 외주화 탐색:

- 대신 써달라는 요청 탐지 가능
- 붙여넣기 시도 기록 가능
- 초안 변화와 AI 대화 사이의 관계 관찰 가능
- 최종 글과 과정 로그 비교 가능

비판적 사고 과정 탐색:

- 근거 요청, 출처 메모, 반론 입력, 주장 수정 이벤트를 관찰 가능
- 개요 점검 경고와 이후 수정 여부 확인 가능
- 피드백을 보고 수정했는지 기록 가능

AI 아첨 탐지 벤치마크 준비:

- assistant turn 단위 라벨링 CSV 생성 가능
- assistant responseType과 실제 응답 텍스트를 비교 가능
- 코드북의 아첨 라벨 축은 준비됨

## 14. 현재 구현의 중요한 한계

연구 설계 관점에서 아직 비어 있는 부분은 다음과 같다.

자동 라벨링:

- LLM judge가 비판적 사고, 인지 외주화, 아첨을 자동 라벨링하는 기능은 아직 없다.
- CSV는 라벨을 붙일 수 있는 틀만 제공한다.

실험 조건:

- 과제와 세션에는 `researchCondition`이 저장된다.
- 현재 실제 활성 조건은 `single_group_baseline` 하나이다.
- `understanding_calibration`의 AI 채팅은 `single_group_baseline` 전용 독해 보조 프롬프트를 사용한다.
- `evidence_check`, `challenge`, `explanation_rich`는 향후 확장을 위해 코드 상수로만 예약되어 있으며 교사 화면에는 노출되지 않는다.
- 학생을 조건에 무작위 배정하는 기능도 없다.

자기평가와 독립 설명 수행 관찰:

- 학생은 AI와 대화한 뒤 수행 예측 문항에 답한다.
- 독립 수행은 네 문제로 나뉘며 한 번에 하나씩만 표시된다.
- 각 문제 제출 후 `problem1_confidence`부터 `problem4_confidence`까지 별도 확신도가 저장된다.
- 플랫폼은 calibration gap, 독립 수행 점수, LLM judge 결과를 자동 계산하지 않는다.

근거 조사:

- 내장 브라우저 검색, RAG, 외부 자료 수집 기능은 없다.
- 현재는 학생이 직접 출처 메모를 입력하는 구조이다.

보안과 운영:

- 평문 비밀번호 기반 파일럿 인증이다.
- 서버 DB, 암호화, 세션 만료, 교사별 권한 분리, 감사 로그는 없다.
- 실제 배포용 개인정보 보호/동의/익명화 플로우는 없다.

디자인 완성도:

- 칸아카데미 Writing Coach의 흐름과 일부 시각 언어를 참고했지만, 동일 제품 수준의 세부 인터랙션과 운영 UX를 완전히 복제한 상태는 아니다.
- 현재 목적은 연구 파일럿용 최소 운영 가능 상태에 가깝다.

## 15. 다음 보완 우선순위

연구 목적에 맞춰 다음 순서가 적절하다.

1. 조건 확장과 배정 설계

   현재 1차 연구는 `single_group_baseline`으로 고정되어 있다. 다음에는 예약 조건을 실제로 활성화할지, 조건별 학생 배정이나 무작위 배정을 넣을지 정해야 한다.

2. 라벨링 파이프라인 설계

   export된 이벤트/대화/문제 답안 행을 입력으로 받아 필요한 코드북 라벨을 붙이는 절차를 별도로 설계한다.

3. 독립 수행 채점 기준 설계

   플랫폼 안에서 자동 채점하지 않고, 연구자가 사후에 적용할 루브릭과 복수 채점자 절차를 정한다.

4. 후속 과제 설계

   필요하다면 지연 과제나 다른 주제의 전이 과제를 별도 세션으로 추가한다.

5. 로그 구조 정교화

   현재도 문제별/학생별 분석은 가능하지만, 대시보드와 export에서 과제별, 반별, 학생별 묶음을 더 명확히 보여주면 후속 분석이 쉬워진다.

6. 개인정보 처리 정리

   파일럿 전 실제 이름 대신 연구 ID 사용, 비밀번호 export 제외 검증, 참여 동의 문구, 데이터 삭제 절차가 필요하다.

## 16. 현재 상태 요약

현재 구현은 “비문학 글쓰기 과제 수행 중 학생-AI 대화와 글쓰기 과정을 수집하는 파일럿 플랫폼”으로는 작동한다.

가장 강한 부분:

- 학생별/과제별 세션 로그
- AI 대화 턴 저장
- 개요, 초안, 붙여넣기, 피드백, 제출 기록
- JSON/CSV export
- 라벨링 코드북과 데이터 사전
- 교사 검토 화면
- 반/학생/교사 계정 관리

아직 연구 플랫폼으로 보완해야 하는 부분:

- 자동 라벨링
- 실험 조건 배정
- 메타인지 측정
- 독립 수행/전이 과제
- 실제 서버 DB와 보안
- 연구 윤리/익명화 플로우

따라서 현재 버전은 최종 연구 플랫폼이라기보다, 대화와 글쓰기 과정 데이터를 실제로 모아보는 1차 파일럿 버전으로 보는 것이 가장 정확하다.
